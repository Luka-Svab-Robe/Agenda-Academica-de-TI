Especificação de Software

Projeto: Agenda Acadêmica para Alunos de TI

1. Introdução

Este documento apresenta a especificação de software para um sistema de "Agenda Acadêmica", focado em alunos de cursos de graduação da área de Tecnologia da Informação. O objetivo é criar uma ferramenta que vá além de uma simples agenda, auxiliando o aluno no gerenciamento de suas disciplinas, atividades e, como diferencial, no cálculo de suas notas e desempenho.

Como um projeto de primeiro semestre, o foco está em definir claramente os requisitos e modelar uma solução simples, coesa e funcional. A especificação segue as etapas fundamentais da engenharia de requisitos e modelagem de dados.

Diferencial Proposto: Além de uma agenda de tarefas (provas, trabalhos), o sistema incluirá um módulo de "Cálculo de Média" por disciplina. O aluno poderá lançar as notas obtidas e os pesos de cada avaliação (P1, P2, Trabalhos), e o sistema calculará automaticamente a média parcial e quanto ele precisa na prova final, ajudando no planejamento de estudos.

2. Listagem de Requisitos

2.1. Requisitos Funcionais (RF)

Descrevem o que o sistema *deve fazer*.

  • RF01: O sistema deve permitir que o usuário (aluno) se cadastre e faça login.
  • RF02: O sistema deve permitir ao aluno cadastrar suas disciplinas (Ex: "Algoritmos", "Engenharia de Software").
  • RF03: O sistema deve permitir ao aluno adicionar atividades a uma disciplina (Ex: "Prova 1", "Trabalho em Grupo").
  • RF04: O aluno deve poder definir dados para a atividade, como data de entrega, prioridade (Baixa, Média, Alta) e tipo (Prova, Trabalho, Tarefa).
  • RF05: O sistema deve exibir um "Dashboard" com as próximas atividades a vencer.
  • RF06: O sistema deve permitir ao aluno marcar uma atividade como "Concluída".
  • RF07 (Diferencial): O aluno deve poder definir os pesos das avaliações de uma disciplina (Ex: P1 - peso 40%, P2 - peso 60%).
  • RF08 (Diferencial): O aluno deve poder lançar a nota obtida em cada atividade/avaliação.
  • RF09 (Diferencial): O sistema deve calcular e exibir a média parcial do aluno na disciplina com base nas notas e pesos.

2.2. Requisitos Não Funcionais (RNF)

Descrevem *como* o sistema deve ser (qualidades e restrições).

  • RNF01 (Usabilidade): A interface deve ser simples, intuitiva e fácil de usar por um aluno com conhecimentos básicos de tecnologia.
  • RNF02 (Plataforma): O sistema deve funcionar como uma aplicação web, acessível por navegadores modernos em desktops e celulares (Design Responsivo).
  • RNF03 (Tecnologia): A aplicação (protótipo) deve ser desenvolvida usando HTML, CSS e JavaScript.
  • RNF04 (Desempenho): As telas principais do sistema devem carregar em menos de 3 segundos em uma conexão de internet padrão.
  • RNF05 (Idioma): O sistema deve ser totalmente em Português (BR).

4. Modelagem de Dados

Para um sistema de 1º semestre, focamos em um modelo simples para armazenar os dados essenciais.

4.1. Modelo Entidade-Relacionamento (MER)

O MER descreve as "coisas" (entidades) que o sistema armazena e como elas se relacionam. Abaixo uma representação visual simplificada usando HTML/CSS.

4.2. Modelo Lógico (Esquema de Tabelas)

O Modelo Lógico traduz o MER para a estrutura de tabelas de um banco de dados relacional.

Tabela: ALUNO

Coluna Tipo de Dado Chave / Observação
id_aluno INTEGER PK (Chave Primária)
nome VARCHAR(100)
email VARCHAR(100) Único
senha VARCHAR(255) (Armazenar hash)

Tabela: DISCIPLINA

Coluna Tipo de Dado Chave / Observação
id_disciplina INTEGER PK (Chave Primária)
id_aluno INTEGER FK (Chave Estrangeira de ALUNO)
nome_disciplina VARCHAR(100)
professor VARCHAR(100)

Tabela: ATIVIDADE

Coluna Tipo de Dado Chave / Observação
id_atividade INTEGER PK (Chave Primária)
id_disciplina INTEGER FK (Chave Estrangeira de DISCIPLINA)
nome_atividade VARCHAR(100)
data_entrega DATE
prioridade VARCHAR(10) ('Baixa', 'Média', 'Alta')
status VARCHAR(10) ('Pendente', 'Concluída')
peso DECIMAL(5,2) Ex: 30.00 (para 30%)
nota DECIMAL(4,2) Ex: 8.50

5. Referências Bibliográficas

Para um projeto de 1º semestre, a base teórica é fundamental. As referências abaixo são clássicos da área de Engenharia de Software e Banco de Dados:

  • PRESSMAN, Roger S. Engenharia de Software: Uma Abordagem Profissional. 8ª ed. Porto Alegre: AMGH, 2016.

    (Referência principal para requisitos funcionais, não funcionais e processos de software.)

  • SOMMERVILLE, Ian. Engenharia de Software. 10ª ed. São Paulo: Pearson Education, 2019.

    (Excelente para entender os diferentes tipos de requisitos e a importância da especificação.)

  • ELMASRI, R.; NAVATHE, S. B. Sistemas de Banco de Dados. 6ª ed. São Paulo: Pearson Education, 2011.

    (Referência clássica para aprender sobre Modelo Entidade-Relacionamento e Modelo Lógico/Relacional.)
